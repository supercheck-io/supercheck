AI SRE release gates protect the read-only investigation agent from regressions before release-candidate and stable deployments.

Default pull-request CI runs deterministic offline evals only. Release-candidate and stable channels must also run seeded live evals and model-graded evals against a non-production tenant.

<Callout type="warn">
Do not run seeded live evals against customer production tenants. Use fake/demo incidents, short telemetry retention, and read-only connector credentials.
</Callout>

## Gate Levels

| Channel | Deterministic eval | Seeded live eval | Model-graded eval |
| --- | --- | --- | --- |
| `local`, `pull_request` | Blocking | Skipped unless enabled | Skipped unless enabled |
| `canary` | Blocking | Advisory when enabled | Advisory when enabled |
| `release_candidate`, `rc` | Blocking | Blocking and required | Blocking and required |
| `stable`, `production` | Blocking | Blocking and required | Blocking and required |

Run the full release gate from `supercheck/app`:

```bash
npm run test:sre-eval:release
```

## Generate Environment Template

Generate a safe placeholder template from the checked-in seed manifest:

```bash
npm run sre-eval:release-env
```

Generate a stable-channel subset:

```bash
npm run sre-eval:release-env -- \
  --channel=stable \
  --fixture-ids=connector-investigation-oss-lab-checkout-degradation
```

Generate a markdown runbook:

```bash
npm run sre-eval:release-env -- --format=markdown
```

The template uses placeholders for secrets. Populate real values in the CI secret store or release environment only.

## Required Variables

| Variable | Required for RC/stable | Notes |
| --- | --- | --- |
| `SRE_EVAL_RELEASE_CHANNEL` | Yes | Use `release_candidate`, `rc`, `stable`, or `production`. |
| `SRE_EVAL_LIVE_ENABLED` | Yes | Must be `true` for RC/stable. |
| `SRE_EVAL_MODEL_GRADE_ENABLED` | Yes | Must be `true` for RC/stable. |
| `SRE_EVAL_BASE_URL` | Yes | Non-production seeded SuperCheck tenant URL. |
| `SRE_EVAL_AUTH_TOKEN` | Yes | Store as a secret. Scope to SRE eval tenant access only. |
| `SRE_EVAL_FIXTURE_IDS` | Recommended | Use a comma-separated subset while seeding fixtures incrementally. |
| `SRE_EVAL_INCIDENT_IDS` | Yes | JSON map from fixture ID to seeded SRE incident ID. |
| `SRE_EVAL_EVALUATED_MODEL_ID` | Yes | The investigation model under release validation. |
| `SRE_EVAL_GRADER_MODEL_ID` | Recommended | Must differ from the evaluated model when set. |
| `SRE_EVAL_MODEL_GRADE_MIN_SCORE` | Optional | Defaults to `0.8`. |

## Seeded Tenant Checklist

- Create a dedicated non-production organization/project for release evals.
- Create fake/demo services such as `checkout-api`.
- Create one SRE incident per selected fixture and keep the incident IDs in the release secret store.
- Configure only read-only connectors with service scope and bounded output limits.
- Use the optional [AI SRE integration lab](/docs/app/deployment/aisre-lab) for Prometheus, Loki, and Tempo seed data when possible.
- Keep connector credentials, API tokens, and model provider credentials out of source control.
- Delete or rotate stale seed data and tokens after release validation.

## Failure Policy

For `release_candidate` and `stable`, seeded live or model-graded failures are release blockers.

Do not bypass these gates by changing the channel to `canary` or disabling live/model evals. Fix the seed data if the environment is stale, or fix the product regression if the seeded evidence is valid.
