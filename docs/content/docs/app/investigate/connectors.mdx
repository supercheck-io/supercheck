Connectors bridge Supercheck's AI SRE layer to your existing operational tools. By providing the AI agent with read-only access to metrics, logs, and events, it can gather the necessary context to investigate incidents, formulate hypotheses, and suggest fixes.

Administrators configure connectors from **Admin → Integrations**. Responders use the configured connectors from incident workspaces and Copilot.

## Supported Connectors

Supercheck currently supports a wide array of read-only direct connectors:

- **Metrics & Observability**: Prometheus, Grafana, Datadog
- **Logs & Search**: Elasticsearch, OpenSearch, Loki
- **Cloud & Infrastructure**: Kubernetes, AWS CloudWatch
- **Error Tracking & Tracing**: Sentry, Grafana Tempo
- **Code & Version Control**: GitHub

All connectors are strictly read-only. For example, the AWS CloudWatch connector only uses least-privilege IAM permissions like `cloudwatch:DescribeAlarms` and `cloudwatch:GetMetricData`.

## Connector Execution Modes

Supercheck executes connectors using two primary models, ensuring your architecture remains secure regardless of whether you self-host or use cloud infrastructure.

### Direct Execution

For cloud-accessible tools (e.g., GitHub, Datadog, Sentry), the Supercheck app queries these services directly, passing the necessary authentication tokens and parameters based on server-side RBAC and strict service scoping.

### Private Agent Mode

For internal services protected behind firewalls (e.g., on-premise Kubernetes clusters, private Elasticsearch nodes, internal Prometheus and Grafana instances, or self-hosted GitHub Enterprise), Supercheck utilizes the **Private Agent**. The private agent supports the implemented read-only connector set: **GitHub, Kubernetes, Prometheus, Grafana, Sentry, Datadog, Loki, Elasticsearch/OpenSearch, AWS CloudWatch, and Grafana Tempo**.

The Private Agent is an outbound-only mode of the existing Supercheck worker image. 
- It uses HTTPS long-polling to fetch query tasks from the Supercheck control plane.
- It executes the queries locally against your internal network resources.
- It submits bounded, sanitized evidence summaries back to the app.

The Private Agent ensures that raw upstream logs, complete trace payloads, or massive Elasticsearch documents remain within your internal network. Only the relevant, summarized evidence generated from the query is transmitted back.

## Connector Configuration & Governance

When configuring connectors, administrators benefit from built-in governance and safety measures:

- **Setup & Search UX**: Each connector provides guided setup hints, example queries, and official documentation links.
- **Bounded Search**: Administrators can perform audited read-only searches directly from the connector configuration interface to validate queries without exposing full access.
- **Credential Validation**: Supercheck automatically verifies your credentials upon saving to ensure successful connectivity.
- **Redaction & Limits**: Built-in safeguards automatically enforce time windows, row limits, and byte limits on every query to protect performance.
- **Service Scope**: Connectors can be scoped to specific services and environments to enforce least-privilege context isolation.
- **SRE Integration Bindings**: Administrators can link outbound notification channels (like PagerDuty or Slack) directly to read-only evidence connectors to automatically surface relevant context.

Services, diagnostic recipes, integrations, and Private Agents are documented in [Admin → AI SRE Setup](/docs/app/admin/ai-sre-setup).
